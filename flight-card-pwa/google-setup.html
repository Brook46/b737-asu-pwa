Google Calendar setup (5 min)

Flight Card needs a one-time OAuth Client ID to read your Google Calendar. All the work below happens in Google Cloud Console on a laptop or desktop browser.

1. Create a Google Cloud project

  1. Go to console.cloud.google.com and sign in with the same Google account that owns your roster calendar.
  2. Top bar → project picker → New Project. Name it anything (e.g. "Flight Card"). Create.

2. Enable the Calendar API

  1. Side menu → APIs & Services → Library.
  2. Search "Google Calendar API" → click it → Enable.

3. Configure the consent screen

  1. Side menu → APIs & Services → OAuth consent screen.
  2. User type: External → Create.
  3. App name: "Flight Card" (or whatever you want). User support email + developer email: your email. Save and continue.
  4. Scopes: skip (you'll request scopes from code). Save and continue.
  5. Test users: add the Google account whose calendar you're reading. Save and continue.
Note: while the consent screen is in "testing" mode, only the listed test-user accounts can sign in. That's fine — it's your account.

4. Create the OAuth Client ID

  1. Side menu → APIs & Services → Credentials.
  2. + Create credentials → OAuth client ID.
  3. Application type: Web application.
  4. Name: "Flight Card PWA" (anything).
  5. Authorised JavaScript origins → + Add URI → enter the origins below, one per row:
    • https://brook46.github.io (the GitHub Pages URL where the PWA is hosted)
    • http://localhost:8095 (only if you want to test locally)
    Important: origin-only — no trailing slash, no path. https://brook46.github.io/ (with slash) and https://brook46.github.io/b737-asu-pwa/… (with path) both fail. Google will then surface the error as "Error 401: invalid_client" which is misleading — the real cause is the origin mismatch.
  6. Authorised redirect URIs → + Add URI → enter the full URL Flight Card runs at:
    • https://brook46.github.io/b737-asu-pwa/flight-card-pwa/
    • http://localhost:8095/ (only if testing locally)
    Redirect URIs are full URLs with the path and trailing slash, unlike Authorised JavaScript origins (which are origin-only). Both lists need to be filled in for the redirect sign-in flow to work.
  7. Create. Copy the Client ID — it ends in .apps.googleusercontent.com.

5. Find your Calendar ID

  1. Open calendar.google.com.
  2. In the left side bar, hover the calendar with your roster → 3 dots → Settings and sharing.
  3. Scroll to Integrate calendar. Copy the value labelled Calendar ID. It usually looks like an email — e.g. abc123@group.calendar.google.com, or just your gmail address for your primary calendar.
Tip: if you want to read your default calendar, you can just use primary instead of the long ID.

6. Paste into Flight Card

  1. Open Flight Card on the iPad → ⚙ Settings.
  2. Paste the Client ID into the OAuth Client ID field.
  3. Paste the Calendar ID (or leave primary) into the Calendar ID field.
  4. Tap Sign in with Google. A Google sign-in popup appears — pick your account and approve the read-only Calendar permission.
  5. Tap Sync now. The latest roster event lands in the data card and the leg switcher appears.
If sign-in shows "App not verified" or "Access blocked": that means your Google account isn't on the test-users list. Go back to step 3 and add it. Once added, retry.

What Flight Card sees

The token grants read-only access to your calendar. Flight Card asks the Calendar API for events in the next 14 days, finds the one whose Notes look like a roster (i.e. contains "Cockpit:" or "Slip details"), and parses that text. Nothing leaves your browser — the call is direct from Flight Card to Google.
